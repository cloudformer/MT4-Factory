# MT4 Factory 系统架构

## 🎯 核心目标

**MT4 Factory 是一个策略工程平台（Strategy Factory），而非交易系统。**

### 系统定位

```
✅ 我们是：
  - 策略生成引擎
  - 策略质量验证平台
  - 策略性能评估工具

❌ 我们不是：
  - 完整的交易平台
  - 账户管理系统
  - MT5全量数据同步工具
```

### 核心工作流

```
策略生成 → 信号生成 → 订单执行 → 交易记录 → 质量验证
   ↓           ↓           ↓           ↓           ↓
Strategy    Signal      Trade      记录盈亏    评估策略质量
```

**目标**: 持续产出高质量策略，通过真实交易数据验证策略有效性。

## 📊 交易记录（Trade Data）

### V1 数据策略

**原则**: 只记录系统执行的交易

```python
# Trade模型
class Trade(Base):
    strategy_id = Column(String(32), nullable=False)  # ✅ 必填
    signal_id = Column(String(32), nullable=False)
    account_id = Column(String(32), nullable=False)
    # ...
```

### 数据来源

```
┌─────────────────────────────────────────┐
│ 系统工作流                               │
│                                         │
│ 1. 策略生成信号                          │
│    Strategy → Signal                    │
│                                         │
│ 2. Execution服务执行信号                 │
│    Signal → MT5 Order → Trade           │
│                                         │
│ 3. 记录交易到数据库                      │
│    Trade.strategy_id = signal.strategy_id│
│    Trade.signal_id = signal.id          │
└─────────────────────────────────────────┘
```

**特点**:
- ✅ 所有交易都有`strategy_id`（不为NULL）
- ✅ 可追溯到具体信号和策略
- ✅ 用于策略性能评估
- ❌ 不记录手动交易
- ❌ 不记录其他EA/工具的交易

### 为什么不同步全量MT5数据？

```
┌─────────────────────────────────────────────────┐
│ 如果同步MT5全量数据（V2方案）                     │
│                                                 │
│ MT5账户的所有交易（1000条）                       │
│   ├── 系统策略执行: 600条                        │
│   ├── 用户手动交易: 300条  ← 无strategy_id       │
│   └── 其他EA/工具: 100条   ← 无strategy_id       │
│                                                 │
│ 问题：                                           │
│ ❌ 需要复杂的匹配逻辑（Comment/Magic Number）     │
│ ❌ 大量数据与策略质量无关                         │
│ ❌ 增加系统复杂度                                │
│ ❌ 偏离核心目标（策略工程）                       │
└─────────────────────────────────────────────────┘
```

**结论**: V1方案足够，专注策略质量验证。

## 🔄 策略生命周期

### 策略状态

```python
class StrategyStatus(str, Enum):
    ACTIVE = "ACTIVE"         # 激活：正在产生信号和交易
    CANDIDATE = "CANDIDATE"   # 候选：停止产生信号（初始状态或停用后）
    ARCHIVED = "ARCHIVED"     # 归档：不再使用
```

### 状态转换

```
┌──────────────────────────────────────────────────────┐
│ 策略生命周期                                          │
│                                                      │
│  创建策略                                             │
│     ↓                                                │
│  CANDIDATE ─────────────────────────┐                │
│     ↑                               │                │
│     │ 停用                        激活                │
│     │                               ↓                │
│  ACTIVE (产生信号和交易) ←───── ACTIVE                │
│     │                                                │
│     │ 归档                                            │
│     ↓                                                │
│  ARCHIVED                                            │
└──────────────────────────────────────────────────────┘
```

### 交易记录中的策略状态

**重要**: 交易记录中的`strategy_status`是**当前状态**，不是**交易时状态**。

```
场景示例：

时间线：
  2024-01-01: 策略STR_001状态=ACTIVE → 产生信号 → 执行交易（+100 USD）
  2024-01-05: 策略STR_001状态=ACTIVE → 产生信号 → 执行交易（-50 USD）
  2024-01-10: 用户停用策略 → 状态=CANDIDATE

查看交易记录（2024-01-15）：
  ├── Ticket: 123 | 策略: STR_001 | 状态: CANDIDATE | 盈亏: +100 USD
  └── Ticket: 124 | 策略: STR_001 | 状态: CANDIDATE | 盈亏: -50 USD
                                    ↑ 当前状态是CANDIDATE

✅ 这是正常的！交易时策略是ACTIVE，现在停用了变成CANDIDATE。
```

### 筛选逻辑

```javascript
// 前端筛选：根据策略的当前状态
if (this.tradeFilters.strategy_status === 'active') {
    // 显示所有当前状态为ACTIVE的策略的交易
    filtered = filtered.filter(t => {
        const strategy = this.strategies.find(s => s.id === t.strategy_id);
        return strategy && strategy.status === 'ACTIVE';
    });
}
```

**用途**:
- 查看**当前激活策略**的历史交易性能
- 查看**候选策略**的历史交易（决策是否激活）
- 查看**归档策略**的历史交易（复盘分析）

## 🎨 Dashboard设计理念

### V1 简化方案

**架构**: 前端筛选 + 透明数据范围提示

```
┌─────────────────────────────────────────┐
│ 后端API: /api/trades                     │
│ - 返回最新50条交易                        │
│ - JOIN Account获取login信息               │
│ - 不做复杂筛选和计算                      │
└───────────────┬─────────────────────────┘
                │
                ↓
┌─────────────────────────────────────────┐
│ 前端筛选                                 │
│ - 按账号ID筛选                           │
│ - 按策略ID筛选                           │
│ - 按策略状态筛选（ACTIVE/CANDIDATE...）   │
│ - 计算筛选结果的盈亏                      │
└───────────────┬─────────────────────────┘
                │
                ↓
┌─────────────────────────────────────────┐
│ 显示                                     │
│ 找到 20 条记录                           │
│ 盈亏(20条): +533.06 USD                  │
│                                         │
│ ℹ️ 盈亏基于当前加载的 50 条数据计算       │
│   （已达加载上限，可能还有更多记录）⚠️    │
└─────────────────────────────────────────┘
```

**优点**:
- ✅ 快速实施（低开销）
- ✅ 响应迅速（前端计算）
- ✅ 透明诚实（明确数据范围）

**适用场景**:
- ✅ 数据量小（<100条交易）
- ✅ 粗略查看策略性能
- ✅ 快速筛选和对比

**限制**:
- ⚠️ 只显示最新50条的筛选结果
- ⚠️ 盈亏是已加载数据的统计，非全量

### 何时升级到V2？

**升级信号**:
- 交易记录超过100条
- 经常看到"已达加载上限"警告
- 需要精确的全量统计
- 需要后端分页和汇总

**V2方案**: 后端筛选 + 分页 + 汇总（详见 `TRADE_PAGINATION_ARCHITECTURE.md`）

## 🎯 总结

### 核心原则

1. **专注策略工程**
   - 产出策略
   - 验证质量
   - 持续优化

2. **简洁架构**
   - V1方案满足需求
   - 避免过度设计
   - 快速迭代

3. **数据透明**
   - 明确数据范围
   - 诚实告知限制
   - 不误导用户

### 系统边界

```
✅ 在范围内：
  - 策略生成与管理
  - 信号评估与执行
  - 系统交易记录与分析
  - 策略性能验证

❌ 不在范围内：
  - 完整账户管理
  - 手动交易追踪
  - MT5全量数据同步
  - 复杂交易平台功能
```

**这是一个策略工厂，不是交易系统。** 🏭
