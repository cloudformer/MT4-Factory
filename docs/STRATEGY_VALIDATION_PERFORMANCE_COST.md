# 历史数据回测性能与成本分析

## 📊 三个阶段的算力和时间成本

### Phase 1：保守起步（3品种×3周期×2年）

#### **数据规模**

```
品种: EURUSD, GBPUSD, USDJPY (3个)
周期: H1, H4, D1 (3个)
历史: 2年

详细计算：
  H1: 24根/天 × 730天 = 17,520根
  H4: 6根/天 × 730天 = 4,380根
  D1: 1根/天 × 730天 = 730根
  ─────────────────────────────────
  单品种: 22,630根
  总计: 22,630根 × 3品种 = 67,890根K线
  
存储空间: ~15 MB (含索引)
```

#### **时间成本**

##### **1. 数据导入（一次性）**

| 操作 | 时间 | 说明 |
|------|------|------|
| **从MT5获取数据** | 2-5分钟 | 网络速度影响 |
| 品种1 (EURUSD) | ~30秒 | 22,630根K线 |
| 品种2 (GBPUSD) | ~30秒 | 22,630根K线 |
| 品种3 (USDJPY) | ~30秒 | 22,630根K线 |
| **写入数据库** | 1-2分钟 | PostgreSQL批量插入 |
| **创建索引** | ~30秒 | 自动优化 |
| **总计（首次导入）** | **3-8分钟** | 一次性操作 |

##### **2. 每日更新数据**

| 操作 | 时间 | 频率 |
|------|------|------|
| 获取最新K线 | ~10秒 | 每天约72根新K线（3品种×3周期×8根/天） |
| 写入数据库 | ~5秒 | 增量更新 |
| **每日总计** | **~15秒** | 自动任务（凌晨2点） |

##### **3. 单次策略回测时间**

```
场景：生成1个策略，使用历史数据评估

步骤：
  1. 从数据库查询数据: ~50ms (3000根H1，有索引)
  2. 加载到DataFrame: ~20ms
  3. 运行回测引擎: ~200-500ms (遍历3000根K线)
  4. 计算性能指标: ~50ms
  ────────────────────────────────────────
  单次回测总时间: 300-600ms (0.3-0.6秒)
```

**对比模拟数据**：
- 模拟数据生成: ~100ms
- 回测: ~200-500ms
- 总计: ~300-600ms

**结论：历史数据和模拟数据回测速度相当！** ✅

##### **4. 批量生成策略时间**

| 策略数量 | 回测时间 | 总耗时 | 说明 |
|---------|---------|--------|------|
| **2个策略** | 0.6秒×2 | ~1.2秒 | 默认生成量 |
| **10个策略** | 0.6秒×10 | ~6秒 | 中等批量 |
| **50个策略** | 0.6秒×50 | ~30秒 | 大批量 |
| **100个策略** | 0.6秒×100 | ~1分钟 | 极限测试 |

**并行优化**（可选）：
```python
# 使用多进程并行回测
import multiprocessing

pool = Pool(processes=4)  # 4核并行
results = pool.map(backtest, strategies)

优化后：
  100个策略: 1分钟 → 15-20秒 (4倍速度)
```

#### **算力成本（Phase 1）**

| 资源 | 需求 | 成本 |
|------|------|------|
| **CPU** | 2核（单线程足够）<br>4核（并行优化） | 本地Mac/VPS即可 |
| **内存** | ~500MB（运行时）<br>峰值1GB | 轻量级 |
| **存储** | 15MB（数据）<br>+50MB（索引+日志） | 几乎可忽略 |
| **网络** | 仅首次导入需要（MT5连接） | 无持续成本 |

**总结：Phase 1 几乎无额外算力成本，本地Mac即可运行。** 💰✅

### Phase 2：中等扩展（10品种×5周期×5年）

#### **数据规模**

```
品种: 10个主流货币对
周期: M15, M30, H1, H4, D1 (5个)
历史: 5年

详细计算：
  M15: 96根/天 × 1,825天 = 175,200根
  M30: 48根/天 × 1,825天 = 87,600根
  H1:  24根/天 × 1,825天 = 43,800根
  H4:  6根/天 × 1,825天 = 10,950根
  D1:  1根/天 × 1,825天 = 1,825根
  ─────────────────────────────────────────
  单品种: 319,375根
  总计: 319,375根 × 10品种 = 3,193,750根K线
  
存储空间: ~500 MB (含索引)
```

#### **时间成本**

##### **1. 数据导入（一次性）**

| 操作 | 时间 | 说明 |
|------|------|------|
| **从MT5获取数据** | 30-60分钟 | 大量数据，分批获取 |
| 每个品种 | ~3-6分钟 | 319,375根K线 |
| **写入数据库** | 10-20分钟 | 批量插入3.2M行 |
| **创建索引** | 5-10分钟 | 多字段索引优化 |
| **总计（首次导入）** | **45-90分钟** | 一次性操作，可后台运行 |

**优化方案**：
```python
# 并行导入（多进程）
from concurrent.futures import ThreadPoolExecutor

def import_symbol(symbol):
    # 导入单个品种
    ...

symbols = ['EURUSD', 'GBPUSD', ...]
with ThreadPoolExecutor(max_workers=4) as executor:
    executor.map(import_symbol, symbols)

优化后: 90分钟 → 20-30分钟 (3-4倍速度)
```

##### **2. 每日更新数据**

| 操作 | 时间 | 频率 |
|------|------|------|
| 获取最新K线 | ~30秒 | 10品种×5周期，约500根新K线/天 |
| 写入数据库 | ~10秒 | 增量更新 |
| **每日总计** | **~40秒** | 自动任务 |

##### **3. 单次策略回测时间**

```
场景：使用5年历史数据回测

数据查询优化（重要）：
  策略通常只需要1个周期（如H1）
  查询: SELECT * FROM historical_bars 
        WHERE symbol='EURUSD' AND timeframe='H1'
        ORDER BY time DESC LIMIT 3000
  
  有索引: ~80ms (从320万行中快速定位)
  无索引: ~5000ms (全表扫描，太慢！)

回测时间：
  1. 数据库查询: ~80ms (有索引)
  2. 加载DataFrame: ~30ms
  3. 运行回测: ~500ms (3000根K线)
  4. 计算指标: ~50ms
  ─────────────────────────────────
  单次回测: ~660ms (0.66秒)
```

**关键：索引非常重要！** 📊

##### **4. 批量生成策略时间**

| 策略数量 | 回测时间 | 总耗时 |
|---------|---------|--------|
| **10个策略** | 0.66秒×10 | ~6.6秒 |
| **50个策略** | 0.66秒×50 | ~33秒 |
| **100个策略** | 0.66秒×100 | ~66秒 (1.1分钟) |
| **500个策略** | 0.66秒×500 | ~5.5分钟 |

**并行优化后**（4核）：
```
100个策略: 66秒 → 17秒
500个策略: 5.5分钟 → 1.4分钟
```

#### **算力成本（Phase 2）**

| 资源 | 需求 | 成本/月 | 说明 |
|------|------|---------|------|
| **CPU** | 4核推荐 | $0 | 本地Mac/VPS均可 |
| **内存** | 2-4GB（运行时） | $0 | 不算大 |
| **存储** | 500MB数据 + 500MB空间 | $0 | PostgreSQL主库 |
| **网络** | 仅每日更新（<1MB/天） | $0 | 可忽略 |

**总结：Phase 2 依然无额外成本，4核Mac轻松应对。** 💰✅

### Phase 3：大规模（30品种×7周期×10年）

#### **数据规模**

```
品种: 30个货币对 + 贵金属 + 指数
周期: M5, M15, M30, H1, H4, D1, W1 (7个)
历史: 10年

详细计算：
  M5:  288根/天 × 3,650天 = 1,051,200根
  M15: 96根/天 × 3,650天 = 350,400根
  M30: 48根/天 × 3,650天 = 175,200根
  H1:  24根/天 × 3,650天 = 87,600根
  H4:  6根/天 × 3,650天 = 21,900根
  D1:  1根/天 × 3,650天 = 3,650根
  W1:  0.14根/天 × 3,650天 = 520根
  ────────────────────────────────────────
  单品种: 1,690,470根
  总计: 1,690,470根 × 30品种 = 50,714,100根K线 (5千万根!)
  
存储空间: ~3-4 GB (原始数据 + 索引)
```

#### **时间成本**

##### **1. 数据导入（一次性）**

| 操作 | 时间 | 说明 |
|------|------|------|
| **从MT5获取数据** | 4-8小时 | 极大量数据 |
| 每个品种 | ~8-16分钟 | 1.69M根K线 |
| **写入数据库** | 2-4小时 | 50M行插入 |
| **创建索引** | 30-60分钟 | 多字段索引 |
| **总计（首次导入）** | **6-12小时** | **建议分批后台运行** |

**优化方案**：
```python
# 1. 分批导入（按品种并行）
with ThreadPoolExecutor(max_workers=8) as executor:
    executor.map(import_symbol, symbols)

# 2. 批量插入优化
session.bulk_insert_mappings(HistoricalBar, batch, render_nulls=True)
session.commit()  # 每10000行提交一次

# 3. 延迟索引创建
# 先导入所有数据（无索引，快）
# 最后一次性创建索引

优化后: 12小时 → 2-4小时 (3-6倍速度)
```

##### **2. 每日更新数据**

| 操作 | 时间 | 频率 |
|------|------|------|
| 获取最新K线 | ~2分钟 | 30品种×7周期，约7000根新K线/天 |
| 写入数据库 | ~30秒 | 增量更新（有索引） |
| **每日总计** | **~2.5分钟** | 自动任务，可接受 |

##### **3. 单次策略回测时间**

```
场景：10年历史数据回测

数据查询性能（关键）：
  查询: WHERE symbol='EURUSD' AND timeframe='H1'
        LIMIT 3000
  
  优化 - PostgreSQL分区表:
    按年分区 (_2024, _2023, ...)
    查询最近3000根只扫描最近表
    时间: ~100ms (优化后)
  
  未优化 (全表扫描):
    时间: ~8000ms (8秒，太慢！)

回测时间（优化后）：
  1. 数据库查询: ~100ms
  2. 加载DataFrame: ~40ms
  3. 运行回测: ~600ms
  4. 计算指标: ~60ms
  ──────────────────────────
  单次回测: ~800ms (0.8秒)
```

**关键优化：必须使用PostgreSQL分区表！** ⚡

##### **4. 批量生成策略时间**

| 策略数量 | 回测时间 | 串行 | 并行(8核) |
|---------|---------|------|-----------|
| **100个策略** | 0.8秒×100 | ~80秒 | ~10秒 |
| **500个策略** | 0.8秒×500 | ~6.7分钟 | ~50秒 |
| **1000个策略** | 0.8秒×1000 | ~13.3分钟 | ~1.7分钟 |
| **AI批量(5000)** | 0.8秒×5000 | ~67分钟 | ~8.4分钟 |

**AI批量生成场景**（关键）：
```
假设：每天AI生成100个候选策略

Phase 1-2 (无并行):
  100个 × 0.6秒 = 1分钟 ✅ 可接受

Phase 3 (无优化):
  100个 × 8秒 = 13分钟 ⚠️ 太慢

Phase 3 (分区+索引):
  100个 × 0.8秒 = 80秒 ✅ 可接受

Phase 3 (分区+并行8核):
  100个 × 0.8秒 ÷ 8 = 10秒 ⭐ 完美
```
