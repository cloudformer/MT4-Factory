# 评估加权方案设计文档

## 📊 背景

策略评估器支持三种评估方式：
1. **合成数据评估** (Synthetic) - 模拟数据快速评估
2. **历史数据评估** (Historical) - 真实历史数据回测
3. **实时数据评估** (Realtime) - 实时行情纸面交易

问题：如何合理地综合多种评估结果？简单平均 vs 加权平均？

## 🎯 加权方案

### 最终采用的权重配置

```python
evaluation_weights = {
    'historical': 0.60,  # 60% - 历史数据最可靠
    'synthetic': 0.20,   # 20% - 合成数据辅助参考
    'realtime': 0.20     # 20% - 实时数据防止短期噪声
}
```

### 动态权重调整规则

| 启用的评估方式 | 权重分配 | 应用场景 |
|---------------|---------|---------|
| **仅 synthetic** | synthetic: 100% | 开发阶段，快速筛选 |
| **synthetic + historical** | historical: 75%, synthetic: 25% | 准实盘前验证 |
| **synthetic + realtime** | synthetic: 60%, realtime: 40% | 实时测试为主 |
| **historical + realtime** | historical: 75%, realtime: 25% | 实盘前最后验证 |
| **全部三个** | historical: 60%, synthetic: 20%, realtime: 20% | 全面评估 |

## 📐 设计理由

### 1. 为什么历史数据权重最高？(60%)

**✅ 优点**:
- 真实市场数据，包含真实的价格波动、流动性、市场微观结构
- 可以验证策略在历史不同市场环境下的表现
- 最接近实盘情况（假设未来类似历史）

**数据可靠性排序**: Historical > Realtime > Synthetic

### 2. 为什么合成数据权重较低？(20%)

**⚠️ 局限性**:
- 模拟数据过于理想化，缺少真实市场的"脏数据"
- 无法模拟突发事件、流动性枯竭、滑点等真实情况
- 随机游走可能与真实市场特征不符

**作用**: 快速筛选，初步验证逻辑，但不应主导最终决策

### 3. 为什么实时数据权重较低？(20%)

**⚠️ 噪声风险**:
- 60分钟的实时测试样本量太小
- 可能遇到特殊行情（重大新闻、假期前后）
- 短期表现不代表长期表现

**关键洞察**: **防止短期噪声** - 这是用户提出的核心考量！

如果实时测试权重过高（如50%），可能导致：
```
历史回测: 72分 (3000根K线，稳定)
实时测试: 85分 (1小时，遇到单边行情) ← 噪声
综合评分: 78.5分 ← 被短期噪声拉高，误导决策
```

合理权重(20%)可以缓解这个问题：
```
历史回测: 72分 × 60% = 43.2
合成数据: 68分 × 20% = 13.6
实时测试: 85分 × 20% = 17.0
综合评分: 73.8分 ← 更稳健
```

## 📈 实际应用效果

### 场景1：历史表现优秀，实时测试一般

```
历史回测: 75分 (真实数据，可靠)
合成数据: 68分 (模拟数据，辅助)
实时测试: 60分 (短期波动，噪声)

加权平均:
= 75 × 0.6 + 68 × 0.2 + 60 × 0.2
= 45 + 13.6 + 12
= 70.6分

结论: 综合评分仍然较高(70.6)，历史数据为主导 ✅
```

### 场景2：历史表现一般，实时测试优秀

```
历史回测: 60分 (真实数据，表现平平)
合成数据: 65分 (模拟数据，辅助)
实时测试: 85分 (短期幸运，噪声！)

加权平均:
= 60 × 0.6 + 65 × 0.2 + 85 × 0.2
= 36 + 13 + 17
= 66分

结论: 综合评分不会被短期噪声误导(66而非78) ✅
```

### 场景3：所有评估一致性高

```
历史回测: 72分
合成数据: 70分
实时测试: 74分

加权平均:
= 72 × 0.6 + 70 × 0.2 + 74 × 0.2
= 43.2 + 14 + 14.8
= 72分

一致性: high (差异仅4分)
结论: 策略稳定，值得信赖 ✅✅
```

## 🔍 对比：简单平均 vs 加权平均

| 场景 | 简单平均 | 加权平均 | 更优方案 |
|------|---------|---------|---------|
| 历史75, 实时60 | 67.5 | 70.6 | ✅ 加权（历史更可靠） |
| 历史60, 实时85 | 72.5 | 66.0 | ✅ 加权（避免噪声） |
| 历史72, 实时74 | 73.0 | 72.0 | ⚖️  差异不大 |

**结论**: 加权平均在数据质量差异较大时，能做出更合理的判断。

## 🎨 权重调整建议

当前配置是平衡方案，可根据实际情况调整：

### 保守方案（更重视历史数据）
```python
evaluation_weights = {
    'historical': 0.70,  # 提高到70%
    'synthetic': 0.15,   # 降到15%
    'realtime': 0.15     # 降到15%
}
```

适用于：风险厌恶型投资者，要求历史回测非常可靠

### 激进方案（更重视实时表现）
```python
evaluation_weights = {
    'historical': 0.50,  # 降到50%
    'synthetic': 0.20,   # 保持20%
    'realtime': 0.30     # 提高到30%
}
```

适用于：追求实时适应性，愿意承担短期波动风险

### 均衡方案（当前默认）
```python
evaluation_weights = {
    'historical': 0.60,  # 历史为主
    'synthetic': 0.20,   # 辅助参考
    'realtime': 0.20     # 防止噪声
}
```

适用于：大多数场景，平衡可靠性和适应性

## 📝 总结

### 核心设计原则

1. **历史数据优先** - 真实数据最可靠，权重最高(60%)
2. **防止短期噪声** - 实时数据权重较低(20%)，避免误导
3. **合成数据辅助** - 用于快速筛选，不主导决策(20%)
4. **动态权重调整** - 根据实际启用的评估方式自动调整

### 用户反馈验证

✅ 用户关键洞察：
> "实时数据比重不高为了防止噪声"

这个设计完全符合用户的核心需求！
